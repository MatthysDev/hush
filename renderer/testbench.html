4 façons de couper Discord — teste laquelle marche sur un vrai call.

La vraie API : on parle au socket IPC local de Discord et on pose l'état mute via SET_VOICE_SETTINGS. Aucune touche synthétique.

Créer l'app : discord.com/developers/applications → New Application → Copie l'Application ID (= client_id) et un Client Secret (onglet OAuth2) → dans OAuth2, ajoute la redirect http://localhost. Puis Connecter (Discord affichera une popup d'autorisation). ⚠️ Le scope rpc.voice.write est whitelisté par Discord : si refusé, le log le dira.

On envoie le raccourci mute de Discord au niveau HID (comme un vrai clavier), pas au niveau session comme nut-js. Combo utilisé = celui configuré dans Hush : …

1er lancement : macOS va demander d'autoriser hush-hid dans Réglages › Confidentialité › Accessibilité (l'injection d'événements y est soumise). Active-le puis réessaie. Discord doit avoir sa keybind mute réglée sur ce combo.

On clique le bouton mute de Discord via l'API Accessibilité (comme VoiceOver), pas de touche du tout.

1er lancement : autorise hush-ax dans Réglages › Confidentialité › Accessibilité. Discord doit être lancé avec sa fenêtre ouverte. Si “Toggle” ne trouve rien, clique “Lister les boutons” pour voir les libellés réels.

Le vrai fix audio isole l'entrée de Discord du micro de Wispr — ça demande un device virtuel (BlackHole/Loopback). Ici : détection + un test dégradé.

⚠️ Le test dégradé coupe l'entrée système → ça coupe aussi Wispr. C'est la limite sans device virtuel. Installe brew install blackhole-2ch pour le vrai test, puis relance la détection.
